# Stack Dependencies - Basic Example

Based on [RFC #5663](https://github.com/gruntwork-io/terragrunt/issues/5663) - Stack Dependencies via Autoinclude.

## What this demonstrates

A stack with two units where one unit depends on the other:

- `unit_w_outputs` - produces outputs `val = "Hello!"` and `id = "unit-w-outputs-12345"`
- `unit_w_inputs` - consumes `val` via a `dependency` block wired through `autoinclude`

The `autoinclude` block in the stack file generates a `terragrunt.autoinclude.hcl` file during `stack generate` that is automatically merged into the unit configuration at parse time.

## Directory structure

```
basic/
  catalog/
    units/
      unit-w-outputs/     # Produces output "val"
        main.tf
        terragrunt.hcl
      unit-w-inputs/      # Consumes val from unit-w-outputs
        main.tf
        terragrunt.hcl
  live/
    terragrunt.stack.hcl  # Stack definition with autoinclude
```

## Stack file

```hcl
locals {
  env = "test"
}

unit "unit_w_outputs" {
  source = "../catalog/units/unit-w-outputs"
  path   = "unit-w-outputs"
}

unit "unit_w_inputs" {
  source = "../catalog/units/unit-w-inputs"
  path   = "unit-w-inputs"

  autoinclude {
    dependency "unit_w_outputs" {
      config_path = unit.unit_w_outputs.path

      mock_outputs_allowed_terraform_commands = ["validate", "plan"]
      mock_outputs = {
        val = "fake-val"
      }
    }

    inputs = {
      val = dependency.unit_w_outputs.outputs.val
    }
  }
}
```

## How to run

### Prerequisites

- Terragrunt built from the `prototype-stack-dependecnies` branch
- OpenTofu (>= 1.6.0) or Terraform (>= 0.12.0) installed

### Step 1: Generate the stack

```bash
cd live
terragrunt stack generate --experiment stack-dependencies
```

This creates `.terragrunt-stack/` with generated units:

```
live/.terragrunt-stack/
  unit-w-outputs/
    main.tf
    terragrunt.hcl
    .terragrunt-stack-manifest
  unit-w-inputs/
    main.tf
    terragrunt.hcl
    terragrunt.autoinclude.hcl   # <-- Generated by autoinclude
    .terragrunt-stack-manifest
```

### Step 2: Inspect the generated autoinclude file

```bash
cat .terragrunt-stack/unit-w-inputs/terragrunt.autoinclude.hcl
```

Expected output:

```hcl
# Generated by Terragrunt. Do not edit manually.
dependency "unit_w_outputs" {
  config_path                             = "../unit-w-outputs"
  mock_outputs_allowed_terraform_commands = ["validate", "plan"]
  mock_outputs = {
    val = "fake-val"
  }
}

inputs = {
  val = dependency.unit_w_outputs.outputs.val
}
```

Key points:
- `config_path` is **relative** (`../unit-w-outputs`, not absolute)
- `mock_outputs` are preserved from the stack file
- `inputs.val` references `dependency.unit_w_outputs.outputs.val` (deferred, not evaluated at generation time)

### Step 3: Plan (uses mock outputs)

```bash
terragrunt run --all --experiment stack-dependencies -- plan
```

Expected behavior:
1. `unit_w_outputs` plans first (no dependencies) — shows `+ local_file.output_marker`
2. `unit_w_inputs` plans second — uses **mock outputs** (`fake-val`) since `unit_w_outputs` is not yet applied

Verify in the plan output:
```
[.terragrunt-stack/unit-w-outputs] tofu: Plan: 1 to add, 0 to change, 0 to destroy.
[.terragrunt-stack/unit-w-outputs] tofu: Changes to Outputs:
[.terragrunt-stack/unit-w-outputs] tofu:   + val = "Hello!"

[.terragrunt-stack/unit-w-inputs]  tofu: Plan: 1 to add, 0 to change, 0 to destroy.
[.terragrunt-stack/unit-w-inputs]  tofu:   + content = "Received: fake-val"   # <-- mock output used

Run Summary  2 units
   Succeeded    2
```

### Step 4: Apply (real outputs flow between units)

```bash
terragrunt run --all --experiment stack-dependencies -- apply -auto-approve
```

Expected behavior:
1. `unit_w_outputs` applies first — creates `output.txt`, produces `val = "Hello!"`
2. `unit_w_inputs` applies second — receives **real output** `val = "Hello!"` from the dependency

Verify in the apply output:
```
[.terragrunt-stack/unit-w-outputs] tofu: Apply complete! Resources: 1 added.
[.terragrunt-stack/unit-w-outputs] tofu: Outputs:
[.terragrunt-stack/unit-w-outputs] tofu:   val = "Hello!"

[.terragrunt-stack/unit-w-inputs]  tofu: Apply complete! Resources: 1 added.
[.terragrunt-stack/unit-w-inputs]  tofu: Outputs:
[.terragrunt-stack/unit-w-inputs]  tofu:   received_val = "Hello!"   # <-- real output used!
```

### Step 5: Validate the dependency output was used

Verify `unit_w_inputs` received the real value from `unit_w_outputs`:

```bash
# Check the file created by unit_w_inputs — should contain the REAL output, not mock
cat .terragrunt-stack/unit-w-inputs/.terragrunt-cache/*/input.txt
# Expected: Received: Hello!

# Check terraform output directly
cd .terragrunt-stack/unit-w-inputs
terragrunt run --experiment stack-dependencies -- output -json
# Expected: {"received_val":{"sensitive":false,"type":"string","value":"Hello!"}}
cd ../..
```

You can also verify the outputs unit produced:

```bash
cd .terragrunt-stack/unit-w-outputs
terragrunt run --experiment stack-dependencies -- output -json
# Expected: {"id":{"...","value":"unit-w-outputs-12345"},"val":{"...","value":"Hello!"}}
cd ../..
```

### Step 6: Plan again (validates real outputs used instead of mocks)

```bash
terragrunt run --all --experiment stack-dependencies -- plan
```

Now that both units are applied, `unit_w_inputs` should show **no changes** and reference the real output:

```
[.terragrunt-stack/unit-w-inputs]  tofu: No changes. Your infrastructure matches the configuration.
```

If `val` had changed (e.g. you modified `unit_w_outputs` to output `"Goodbye!"`), the plan would show:

```
[.terragrunt-stack/unit-w-inputs]  tofu:   ~ content = "Received: Hello!" -> "Received: Goodbye!"
```

### Step 7: Destroy (reverse order)

```bash
terragrunt run --all --experiment stack-dependencies -- destroy -auto-approve
```

Terragrunt destroys in reverse dependency order:
1. `unit_w_inputs` destroyed first (it depends on `unit_w_outputs`)
2. `unit_w_outputs` destroyed second (nothing depends on it)

```
[.terragrunt-stack/unit-w-inputs]  tofu: Destroy complete! Resources: 1 destroyed.
[.terragrunt-stack/unit-w-outputs] tofu: Destroy complete! Resources: 1 destroyed.

Run Summary  2 units
   Succeeded    2
```

### Step 8: Clean up

```bash
terragrunt stack clean
```

This removes the `.terragrunt-stack/` directory entirely.

## Summary of output flow

| Stage | unit_w_inputs receives | Source |
|-------|----------------------|--------|
| `plan` (before apply) | `fake-val` | `mock_outputs` in autoinclude |
| `apply` | `Hello!` | Real `terraform output` from `unit_w_outputs` |
| `plan` (after apply) | `Hello!` | Real state from `unit_w_outputs` |
| `destroy` | Destroyed first | Reverse dependency order |

## How it works

1. **Stack generate** parses `terragrunt.stack.hcl` in two passes:
   - Pass 1: Extract unit/stack names and paths (autoinclude body captured as `hcl.Body` via remain)
   - Pass 2: Resolve `unit.unit_w_outputs.path` to the absolute generated path, evaluate autoinclude dependency `config_path`

2. **File generation** writes `terragrunt.autoinclude.hcl` with:
   - Dependency blocks with relative `config_path` and preserved `mock_outputs`
   - Inputs copied from the AST with `dependency.*` references preserved verbatim

3. **Config parsing** auto-merges `terragrunt.autoinclude.hcl` into the unit config using deep merge (autoinclude wins on conflicts)

4. **DAG construction** builds the dependency graph from the merged config, ordering units correctly for both apply and destroy

5. **Dependency resolution** at runtime:
   - During `plan` (no state yet): uses `mock_outputs` because `mock_outputs_allowed_terraform_commands` includes `"plan"`
   - During `apply`: fetches real outputs from `unit_w_outputs` via `terraform output`
