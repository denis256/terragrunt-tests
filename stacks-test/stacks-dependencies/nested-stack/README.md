# Stack Dependencies - Nested Stack Example

Based on [RFC #5663](https://github.com/gruntwork-io/terragrunt/issues/5663) - Stack Dependencies via Autoinclude.

## What this demonstrates

A sibling `unit` depending on a `unit` that lives inside a nested `stack`.

The unresolved question from the RFC sketch was whether the dependency `config_path` should be written as `stack.foo.path` (the stack root) or `stack.foo.<unit_name>.path` (a unit-by-name inside the stack). The prototype resolves it as **`stack.<stack_name>.<unit_name>.path`** for fine-grained references. The parent stack file parses the child stack's `terragrunt.stack.hcl`, discovers its units, and exposes each one as `stack.<stack_name>.<child_unit_name>` in the eval context (see `BuildComponentRefMap` and `DiscoverStackChildUnits` in `internal/hclparse/stack.go`). `stack.foo.path` still works and refers to the stack root.

- `stack "foo"` - nested stack containing `unit "provider"`
- `unit "bar"` - sibling unit that depends on `stack.foo`'s `provider`

## Directory structure

```
nested-stack/
  catalog/
    stacks/
      foo/
        terragrunt.stack.hcl   # nested stack: defines unit "provider"
    units/
      provider/                # produces output "val"
        main.tf
        terragrunt.hcl
      bar/                     # consumes provider.val
        main.tf
        terragrunt.hcl
  live/
    terragrunt.stack.hcl       # uses stack "foo" + unit "bar"
```

## Stack files

`live/terragrunt.stack.hcl`:

```hcl
stack "foo" {
  source = "../catalog/stacks/foo"
  path   = "foo"
}

unit "bar" {
  source = "../catalog/units/bar"
  path   = "bar"

  autoinclude {
    dependency "provider" {
      config_path = stack.foo.provider.path

      mock_outputs_allowed_terraform_commands = ["validate", "plan"]
      mock_outputs = {
        val = "fake-val"
      }
    }

    inputs = {
      val = dependency.provider.outputs.val
    }
  }
}
```

`catalog/stacks/foo/terragrunt.stack.hcl`:

```hcl
unit "provider" {
  source = "../../units/provider"
  path   = "provider"
}
```

## How to run

```bash
cd live
terragrunt stack generate --experiment stack-dependencies
terragrunt run --all --experiment stack-dependencies -- plan
terragrunt run --all --experiment stack-dependencies -- apply -auto-approve
```

Expected generated layout:

```
live/.terragrunt-stack/
  foo/
    provider/
      main.tf
      terragrunt.hcl
      .terragrunt-stack-manifest
  bar/
    main.tf
    terragrunt.hcl
    terragrunt.autoinclude.hcl
    .terragrunt-stack-manifest
```

Expected `terragrunt.autoinclude.hcl` for `bar`:

```hcl
# Generated by Terragrunt. Do not edit manually.
dependency "provider" {
  config_path                             = "../foo/provider"
  mock_outputs_allowed_terraform_commands = ["validate", "plan"]
  mock_outputs = {
    val = "fake-val"
  }
}

inputs = {
  val = dependency.provider.outputs.val
}
```

Key points:
- `config_path` is rendered **relative** to `bar`'s generated path (`../foo/provider`)
- `stack.foo.provider.path` resolves at generation time to the nested unit's absolute generated path, then is rewritten relative to `bar`
- Apply order: `foo/provider` first, then `bar`
